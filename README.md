# Pathway

> **The mentor who has been at UCLA for 10 years — on demand, with citations.**
> Built for the SoCal Claude Builder Club hackathon at UCLA, April 2026.

A paper-notebook-styled AI roadmap for first-gen college students. The student intakes a short profile (or uploads a resume), and Claude builds a five-stage plan grounded in real UCLA programs — with outreach emails, cover letters, and cited sources generated on demand, tailored per card.

## 1. The hackathon track we attacked

**Event:** SoCal Claude Builder Club Hackathon · UCLA · April 2026 · 1 day · undergraduate.
**Prizes:** $1,500 cash + $3,000 Anthropic API credits.
**Track focus (self-selected):** First-generation student opportunity discovery — an application where Claude adds real leverage because the work is *synthesis*, not *lookup*.

The track was open-ended. We chose a problem we had lived: one of us is first-generation, with parents whose college experience is from another country. Search engines and generic LLMs can list the 200+ programs at UCLA. What they cannot do is pick **the three things this specific student should do in the next 30 days**. That synthesis — personalized, grounded, time-bounded — is where Claude can be a real mentor.

## 2. The problem we picked

College has hundreds of fellowships, research groups, clubs, advising offices, funded summer programs, and capstone opportunities. Students whose families have been through US higher education get pointed to the right ones by name. Students whose families have not — first-gen, transfer, international, working class, or those whose parents are from elsewhere — miss them.

- The bottleneck is not information. It is synthesis.
- The missing role is not a database. It is a mentor.
- The fix is not another aggregator. It is a system that reasons over *this student's* path and recommends the next three moves, with a real human contact at every step.

## 3. What we built

**`/` (onboarding)** — a hand-drawn ruled-notebook intake. Seven questions: name, year, major status, interests, context (first-gen / transfer / international / pre-med / pre-law / etc.), hours per week, and a short "why." Arrow-key navigation, per-device localStorage, no account, no email. Or — skip the questions entirely and upload a PDF resume; Claude extracts the same seven fields automatically.

**`/pathway` (the roadmap)** — a paper-notebook plan with two independently scrolling columns:
- **Left (the roadmap):** five fixed stages — Direction → Community → Signal → Summer → Year-2 Capstone. Click the dashed prompt on a stage; three choices appear, each generated by Claude specifically for this student. Lock one in and a sticky note drops onto the plan board on the right.
- **Right (the plan board):** five dashed cards that fill in as you lock. Once all five lock, a red "Confirm plan · start executing" button reveals. Confirm, and the UI switches to execution mode — each locked card expands with a todo checklist, and the post-it board shows your five signed stickies.

**Per-card outreach drafting.** When you preview any card — a lab intro, a fellowship application, an advisor meeting — Claude decides if the card involves outreach. If yes, the detail panel surfaces two buttons: *"✉️ draft email"* and *"📄 draft cover letter"*. One click fires a tailored draft specific to that opportunity, referencing the student's name, major, interests, and the card's contact. Drafts save per-node to localStorage.

**Resume → profile → pre-written outreach.** The onboarding welcome screen has a second CTA: "upload resume (pdf)". Claude parses the PDF, maps it into the onboarding state, and offers to pre-write a generic research intro email, a cover letter template, and a cold outreach email before the student even sees the roadmap.

## 4. How Claude is used (five separate seams)

1. **`/api/expand-node` — the core roadmap generator.**
   Stage-aware system prompt (`lib/claude.ts`). Each call receives the student's profile, the full path trace of prior lock-ins, and the `stage_key`. Claude returns three options with `path_tag`, `eyebrow`, `why_this`, `why_now`, `todos`, `cites`, and an optional `human_contact`. Prompt caching on the static `<stage_guidance>` + `<output_schema>` blocks via Anthropic's `cache_control: ephemeral`, so each expansion pays only for the novel profile + stage context. Server-side Zod validation rejects malformed responses and drops nodes with broken cite URLs. If the response is missing or invalid, a curated per-stage fallback (`data/ucla/stage_fallbacks.json`) keeps the demo running.

2. **`/api/resume-extract` — resume → structured profile.**
   Anthropic document-input (base64 PDF) + a strict extraction prompt (`app/api/resume-extract/route.ts`). Output is the exact onboarding `state.answers` shape: name, year, major status, interests (up to 3), background flags (first-gen / transfer / etc.), hours per week, and a first-person "why." Rules forbid fabricating credentials — only what the PDF states.

3. **`/api/should-offer-outreach` — a yes/no relevance gate.**
   Before showing the outreach widget on a card, Claude is asked a small, bounded question: does this step involve emailing a human (PI, advisor, hiring manager)? Response is a single JSON `{offer: boolean, reason: string}`. Client-side has a keyword heuristic that runs instantly; the API call refines the reason asynchronously. Failure mode is **fail-open** — if the API is slow or down, the widget still appears and the user can still choose to draft. The gate exists to *remove noise*, not to block access.

4. **`/api/draft-outreach` — per-card and generic outreach.**
   Accepts `{profile, card?, kind?}`. With a card, drafts are tailored: subject line references the specific opportunity, the body addresses the card's contact by name and role, and the closing references the card's stated deadline. Without a card (the welcome-page path), returns three generic-but-tailored drafts. `kind` selects `email`, `cover_letter`, or `auto` (returns everything).

5. **`/api/draft-message` — short message generator (shared helper).**
   Utility used by the executing-mode card to draft quick messages for todos like "email Prof X to confirm office hours." Same SDK, tighter prompt.

Every Claude call is abortable (`AbortController`) with a 10-second timeout (`AbortSignal.timeout`), and every response is validated with Zod before it touches the UI.

## 5. Technical architecture

**Stack:**
- **Next.js 16 App Router** (Turbopack dev)
- **TypeScript strict**
- **Tailwind v4** CSS-first tokens
- **Zustand** for state (chain model for pathway; kept for backend API plumbing)
- **Zod** for every API + data contract
- **Anthropic SDK** — `claude-opus-4-7` for extraction + relevance + drafting; `claude-sonnet-4-6` for high-throughput expand-node
- **Vanilla HTML/CSS/JS** for the UI (see § 6 for why)

**Frontend shape — unusual for a Next.js app:**
The onboarding and pathway pages are **not React components**. They are standalone self-contained HTML files served as static assets from `public/`. Next.js rewrites map `/` → `/onboarding.html` and `/pathway` → `/pathway.html`. This lets us ship the exact designer-approved UX from the Claude Design handoff without a lossy React port. Data flows between the two pages through `localStorage['pathway.onboarding.v3']`. The HTML pages fetch the Next.js API routes directly via `fetch('/api/...')`.

**Backend shape — five stage-aware routes:**
- `POST /api/expand-node` — generate 3 children for a stage
- `POST /api/resume-extract` — PDF → structured profile
- `POST /api/should-offer-outreach` — yes/no gate on outreach widget
- `POST /api/draft-outreach` — email / cover letter / cold outreach drafts
- `POST /api/draft-message` — short message helper

**Safety + testing:**
- 74 passing Vitest unit tests across `tests/` — schema validation, pathway store invariants, claude prompt builder, fallback synth, filter rules, freehand determinism, corpus validator.
- Prompt caching via `cache_control: { type: 'ephemeral' }`.
- Each API route has a request timeout and abort-on-client-disconnect path.
- `.env.local` gated — if `ANTHROPIC_API_KEY` is missing, the fallback path renders and the UI still works.

## 6. The design journey — three pivots

This is part of why the project is a lesson in "ship what the user has already tested, not what you can re-invent."

**Pivot 1 — form triage to branching tree (spec phase).**
Started as "Opportunity Triage Caseworker": form → 3 moves + calendar + emails. Rev 2 chose the branching tree instead. Higher build risk, real differentiator.

**Pivot 2 — full SVG tree to paper notebook (implementation phase).**
We first built a dagre-laid SVG tree with hand-drawn edges. The user's Claude Design handoff came back with a radically different UX: a paper notebook where each stage was a row, stickies dropped on lock-in, and the right panel carried all the citation detail. We rewrote the pathway UI around the notebook aesthetic — rough.js + perfect-freehand, custom CSS module, chain-model store.

**Pivot 3 — React port to verbatim HTML handoff (final phase).**
Halfway through, the user delivered a *second* design handoff (`Scroll directions v2.html`, 127KB of vanilla JS + CSS) with yet another set of features: executing mode, per-stage sticky placement on a separate board, confirm-plan ready state, detail panel with icon meta rows. Rather than re-port manually and lose fidelity, we scrapped the React frontend entirely and served the HTML files verbatim from `public/`, wiring them to the backend. Faster and pixel-perfect.

The backend survived all three pivots untouched. The schema, store, fallback data, filter, and Claude prompts are the stable foundation.

## 7. Demo walkthrough

**The 90-second script:**

1. Land on `/`. Ruled notebook, "Let's figure out your pathway."
2. Press `→`. "What should I call you?" Type "Maya."
3. `→ → → →` through year, major status, interests, context. Select first-gen. Slider to 8 hrs/week. Skip why.
4. Review screen, edit any row with the ✎ icon, hit "build my plan."
5. Land on `/pathway`. Left column shows "YOU ARE HERE · Maya · Freshman · First-gen · CS + AI/ML curious." Right shows 5 empty dashed plan cards.
6. Click the dashed prompt for Stage 1. Loading state briefly. Three Claude-generated direction choices appear. Click one — the right panel loads a detail card: eyebrow, title, meta rows (when / effort / deadline), why-this-fits, what-it-looks-like, sources with clickable UCLA links, and — because this card involves emailing a PI — a collapsed "draft email" widget.
7. Click "✉️ draft email." A research intro email pre-filled with Maya's name, her AI/ML interest, and the specific faculty contact renders inline.
8. Click "Lock it in ✓." The sticky note drops and writes itself. Stage 1 appears on the plan board. Stage 2's dashed prompt unfurls below.
9. Repeat through all five stages. Final board renders "5 stages · ready." Red "Confirm plan · start executing" button fires. Execution mode: each card shows a todo checklist; stickies signed `— M ✓` land on the board.

**The second-persona moment — adaptivity:**
Reset. Start over. This time the persona is pre-med humanities. Upload a resume (or just type answers). Stage 1's three direction choices are now about **declaring Human Biology**, **pre-med gap-year advising**, and **MCAT prep timelines** — not CS tracks. Same backend. Different profile. Different plan.

## 8. Adaptivity — why a doctor and a SWE get different plans

Claude's `<student_context>` block (`lib/claude.ts`) carries **every** signal from onboarding:

```
year, major_category, hours_per_week, first_gen, aid_status, mode,
interests, background (pre-med, pre-law, transfer, international, ...),
name, end_goal
```

The `stage_key` guides what kind of options to generate. The `parent_path_tag` forces coherence — stage 2's community options extend stage 1's direction. The `path_trace` of prior locks gives Claude the student's actual path, not a hypothetical.

Concretely:
- A first-gen CS student sees: ACM AI, AAP peer learning, URFP research, Hack on the Hill, SURP.
- A pre-med humanities student sees: Human Biology declaration, PSA peer advising, volunteer MCAT prep, hospital shadowing, summer research in public health.
- A transfer student sees: transfer-specific orientation programs, bridge fellowships with shorter runways, fast-track research entries.

No code branches on persona. All adaptation is in the prompt — which is exactly where an LLM should handle variance.

## 9. AI ethics and guardrails

**Claude is a triage tool, not an oracle.**

1. **Grounded in named institutions.** Every node must cite at least one UCLA source URL. Fallback data is hand-curated with real links. Citations are rendered as real anchor tags with `target="_blank"` + `rel="noopener noreferrer"` — the student can verify every recommendation in one click. We fixed 57 placeholder `href="#"` links before ship to make this literally true.

2. **Every leaf hands off to a human.** Detail panels expose the actual program coordinator, advisor, or faculty PI when the opportunity has one. The outreach drafts are *starting points the student edits* — never automatic sends.

3. **Resume extraction refuses to fabricate.** The extraction prompt explicitly forbids inventing GPA, credentials, projects, or experiences that aren't in the PDF. If the resume is sparse, fields come back empty rather than hallucinated.

4. **Relevance gate is permissive.** The `should-offer-outreach` check can suppress the draft widget on irrelevant cards (e.g., "Take CS 33"), but it **fails open**: network error or timeout still shows the widget. We would rather offer a tool the student didn't need than hide one they did.

5. **Prompt-injection defense.** User input (resume text, "why" field) is prefixed and labeled in the prompt. The system prompt tells Claude to treat the student's text as content, not instructions.

6. **Epistemic humility.** When the corpus is thin for a profile (e.g., humanities + directed mode), the route falls back to Claude rather than inventing hardcoded non-STEM seeds. The student gets three real-enough options with the caveat that their specific path has less canonical scaffolding.

7. **Minimal persistence of sensitive fields.** Onboarding answers live only in `localStorage` on the student's device. There is no account, no email, no server-side profile store. The app was designed so "clear your browser data" fully deletes your footprint.

8. **No tracking, no analytics, no third-party scripts.** Only external dependencies at the page level are Google Fonts (Caveat + Kalam) and the roughjs CDN for hand-drawn SVG.

9. **Honest uncertainty.** The UI exposes Claude's dropped-count and any epistemic humility block from the API, so the student sees when the system was less confident.

10. **Fallback safety route.** `/api/fallback` (and a static fallback for the legacy linear view) means demo doesn't break on API failure. More importantly, the student gets *something* even when Claude is down.

11. **No destination lock-in.** The post-it sticky signatures, the `name` in outreach drafts, the path-trace-driven prompt — everything the student needs can be regenerated. Nothing we do is destructive to the student's future options.

**What we did not do, and why it matters.**
We did not build a chatbot. We did not ask Claude to "give advice." We constrained every call to a specific, validated shape — a node, a profile, a yes/no, a pre-formatted draft. Every surface is a *tool*, and every tool ends with the student making a choice.

## 10. Impact and who this helps

**Direct impact:** First-gen UCLA undergrads at any stage of uncertainty. The tool compresses the "first 2 hours of figuring out what's even out there" — the part advisors spend most of their time on — into 5 minutes of self-directed exploration. Students arrive at the real advisor oriented, with better questions, and a draft of the email they're about to send.

**Indirect impact:**
- **Advisors' time is freed** from explain-the-landscape to actually advise on trade-offs.
- **Programs gain more applicants who fit** because the matching is happening up front, not after 50 students pick wrong and drop.
- **Transfer and international students** get a purpose-built path rather than a CS-defaulted one — the backend speaks their context.

**Scalability:**
- The architecture is stage-aware, not UCLA-specific. `data/ucla/` can be swapped for `data/<school>/` with a validator-checked fork (§ 12).
- Non-STEM direction seeds fall back to Claude naturally — no engineering work required to support humanities, social sciences, or arts students.
- The React-free public/ HTML approach means the entire UI is <2MB of static assets; hosting is close to free.

**What we'd measure if this were production:**
- Adoption: % of first-gen students who complete onboarding.
- Conversion: % who send at least one drafted outreach email within 30 days.
- Grounding: % of recommended cards that appear in the student's advisor follow-up notes within 90 days.

## 11. Limitations, risks, and what we acknowledge

- **Claude can be overconfident.** Stage-aware prompts help, but we rely on filters + the human handoff to catch it.
- **The corpus is UCLA-only.** Ported seeds work for other R1s, but every school has its own programs. Fork, don't clone.
- **Outreach drafts still require the student to send.** The tool ends at "here's a draft." If the student doesn't edit and send, the impact is zero.
- **The HTML-handoff approach sacrifices React ergonomics.** We can't easily componentize parts of the UI or test them in isolation. This was the right trade-off for fidelity during the hackathon; a production version would React-ify the pages behind the same backend.
- **Single-device profile.** No sync. Opening on another device restarts.
- **Browser support: Chrome 85+ / Safari 15.4+.** We use `@property --write` and `backdrop-filter` — modern only.
- **English-only prompts.** Internationalization is future work.

## 12. Fork guide — porting to another school

1. Replace `data/ucla/` with `data/<school>/`.
2. Required files:
   - `opportunities.json` — 15-20 curated items + 3-5 advising fallbacks.
   - `first_layer_seeds.json` — 4 seeds (one per major category, at least one discovery-mode).
   - `stage_fallbacks.json` — per-stage backup nodes for all 5 stages.
3. Every record must validate against `CorpusItemSchema` / `FirstLayerSeedSchema` / the stage-node contract in `lib/schemas.ts`. The predev validator (`npm run dev` implicitly, or `npx tsx scripts/validate-data.ts`) fails loudly on malformed data.
4. Update the HTML files' `STAGES` fallback arrays (client-side safety net) to your school's program names — or let the Claude fallback do the work.
5. Tweak `lib/claude.ts` `<stage_guidance>` blocks to reference your school's program names in the example text.
6. Ship.

## 14. Local run

```bash
git clone https://github.com/jxdai2007/Pathway
cd Pathway
cp .env.example .env.local                 # paste ANTHROPIC_API_KEY
npm install
npm run dev
open http://localhost:3000
```

Requires Node 20+. Hard-refresh the browser if HMR serves stale HTML.

**Demo persona:** the HTML files default to Maya (Freshman · First-gen · CS/AI). To reset localStorage and try a different persona, open DevTools → Application → Local Storage → clear `pathway.onboarding.v3`.
